# i18n Audit & Implementation Plan for NYX

## 1. Executive Summary

**Status:** 🔴 **Not Ready for Localization**
**Impact:** High (Requires structural changes in both Frontend and Backend)
**Estimated Effort:** Medium-High (1-2 weeks for full implementation)

The NYX application currently lacks any internationalization (i18n) infrastructure. User-facing strings are hardcoded in English across React components, helper functions, and backend API responses. Making the app translatable requires a coordinated effort to decouple text from logic.

---

## 2. Findings

### 2.1 Frontend (`web/`)

*   **Hardcoded Strings:** Found extensively in all major components.
    *   **Forms:** `AuthForm.tsx` ("Login", "Password", "Email")
    *   **Pages:** `ProfilePage.tsx` ("Security Clearance", "Active"), `ConnectPage.tsx`
    *   **Toasts/Alerts:** `App.tsx` ("Settings", "Logout"), `handleApiError` in `api.ts`.
*   **Missing Libraries:** No i18n libraries (like `react-i18next` or `react-intl`) are installed.
*   **Error Handling:** The `handleApiError` utility returns hardcoded English strings (e.g., "Network error", "Session expired") and directly passes through backend error messages.
*   **Date/Time:** Likely using `Date` methods or basic formatting without locale awareness.

### 2.2 Backend (`server/`)

*   **API Responses:** The backend sends human-readable English error messages (e.g., `{ error: "Invalid credentials" }`).
*   **Tight Coupling:** The frontend relies on these backend strings for UI display, making it impossible to translate error messages without changing the backend API contract.

## 3. Implementation Plan

### Phase 1: Foundation (Frontend)

1.  **Install Dependencies:**
    ```bash
    pnpm add i18next react-i18next i18next-browser-languagedetector i18next-http-backend
    ```
2.  **Directory Structure:**
    Create a standard locale structure:
    ```
    web/src/
    ├── i18n.ts          # Initialization config
    └── locales/
        ├── en/
        │   ├── common.json
        │   ├── auth.json
        │   ├── chat.json
        │   └── settings.json
        ├── id/
        │   └── ...
        └── ...
    ```
3.  **Initialization:** Configure `i18next` in `web/src/i18n.ts` and import it in `main.tsx`.

### Phase 2: Refactoring (Frontend)

1.  **Component Audit:** Systematically go through `src/components` and `src/pages`.
2.  **String Extraction:**
    *   Replace `<span>Settings</span>` with `<span>{t('nav.settings')}</span>`.
    *   Move the English string to `web/src/locales/en/common.json`.
3.  **Utilities:**
    *   Refactor `handleApiError` to return error *codes* or translation keys instead of raw strings.
    *   Update `toast` calls to use translated strings.

### Phase 3: Backend Standardization

1.  **Error Codes:**
    *   Modify API endpoints to return stable error codes.
    *   *Before:* `res.status(400).json({ error: "User not found" })`
    *   *After:* `res.status(400).json({ code: "USER_NOT_FOUND", message: "User not found" })`
2.  **Frontend Mapping:**
    *   Update the frontend to listen for `code` and map it to `t('errors.USER_NOT_FOUND')`.

### Phase 4: Automation & Quality

1.  **Linting:** Add `eslint-plugin-i18next` to prevent new hardcoded strings.
2.  **Extraction Tools:** Set up `i18next-parser` to automatically find missing keys.
